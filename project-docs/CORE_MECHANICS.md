# Core Mechanics - new_tax

Документ для быстрого погружения в проект. Читается за 10 минут - после этого `how-calculation-works.md` и код становятся понятными без дополнительных вопросов.

---

## Что это и зачем

Калькулятор НДФЛ для России. Считает, сколько налогов удерживается с зарплаты и сколько взносов платит работодатель - помесячно, с учётом прогрессивной шкалы 2025 года.

Три аудитории: обычный сотрудник (сколько я плачу), HR (сколько стоит сотрудник), разработчик-интегратор (API).

---

## Архитектура: два сервиса

```
[Browser] → [Web / BFF :8080] → [gRPC Backend :50051] → [Redis]
```

**Backend** (`cmd/main.go`, `internal/`) - Go-сервис. Принимает gRPC-запросы, считает налог, кэширует в Redis. Весь расчёт здесь.

**Web / BFF** (`web/`) - отдельный Go-сервис. Рендерит HTML-шаблоны, проксирует запросы на backend через gRPC. Никакой бизнес-логики - только отображение.

**Redis** - кэш результатов расчёта. При недоступности сервис продолжает работать (graceful degradation).

**Контракт** - `docs/grpc/tax.proto`. Из него генерируется `gen/grpc/api/`. Менять контракт → запускать `make codegen`.

## Ключевые концепции

### 1. Все деньги - в копейках (`uint64`)

В коде нет `float64` для денег. Никогда. Всё в копейках.

```
100 000 ₽ → 10 000 000 копеек
```

Почему: прогрессивная шкала с порогами - дробные ошибки накапливаются. Целочисленная арифметика даёт детерминированный результат.

Ставки хранятся аналогично - делятся на 1000:
```
22% → 220 (÷1000)    5.1% → 51 (÷1000)
```

### 2. YTD - накопительный итог

Налог не считается каждый месяц отдельно. Алгоритм:

```
месяц N: tax_raw_ytd = прогрессивный_налог(доход_за_N_месяцев)
месячная_дельта_raw = tax_raw_ytd − tax_raw_ytd_прошлого_месяца
месячный_налог = округлить(месячная_дельта_raw)
```

Почему YTD: при переходе через порог шкалы в середине года без накопительного счёта возникает ошибка. YTD исключает её - каждый месяц пересчитывает "налог за весь год" и берёт дельту.

### 3. Две налоговые базы (при наличии северной надбавки)

При северной надбавке (СН) доход делится на две независимые базы с разными шкалами:

- **База A** - оклад (с учётом РК): облагается по **пятиступенчатой** шкале (13–22%)
- **База B** - северная надбавка: облагается по **упрощённой** шкале (13% / 15%)

Округление применяется к каждой базе **отдельно** на месячной дельте.

### 4. Районный коэффициент (РК) - не отдельная база

РК умножается на оклад и становится частью базы A. Это не отдельная база - просто увеличивает оклад.

```
оклад_с_РК = grossSalary × territorialMultiplier / 100
северная_надбавка = оклад_с_РК × (northernCoefficient − 100) / 100
```

## НПД — налог на профессиональный доход (самозанятые)

НПД — отдельный налоговый режим (Федеральный закон № 422-ФЗ). Он принципиально отличается от НДФЛ: нет страховых взносов, другие ставки, другая база.

### Ставки и выбор между ними

Ставка зависит от источника дохода и определяется один раз на уровне запроса:

- **4%** — доход получен от физических лиц (`NpdRateIndividual = 4`)
- **6%** — доход получен от организаций или ИП (`NpdRateLegal = 6`)

Районный коэффициент и северная надбавка на ставку НПД не влияют.

### Регистрационный вычет

При регистрации самозанятый получает бонус 10 000 ₽ (1 000 000 копеек, константа `NpdRegistrationDeduction = 1_000_000`).

Бонус не выдаётся деньгами — он снижает ставку налога каждый месяц, пока не исчерпан:

- При ставке 4%: фактически применяется 3% (снижение на 1 п.п., `bonusRate = 1`)
- При ставке 6%: фактически применяется 4% (снижение на 2 п.п., `bonusRate = 2`)

Алгоритм за каждый месяц:
1. Считается базовый налог: `monthlyGross × rate / 100`
2. Считается экономия за месяц: `monthlyGross × bonusRate / 100`
3. Экономия вычитается из оставшегося бонуса; остаток переходит на следующий месяц
4. Из базового налога вычитается использованная часть бонуса
5. Результат округляется через `RoundTaxAmount()`

Пример: доход 50 000 ₽/мес, ставка 4%, вычет активен.
- Базовый налог: 50 000 × 4% = 2 000 ₽
- Экономия за месяц: 50 000 × 1% = 500 ₽
- Фактический налог: 2 000 − 500 = 1 500 ₽
- Бонус расходуется по 500 ₽/мес, исчерпается через 20 месяцев (10 000 ₽ ÷ 500 ₽/мес)

### Лимит дохода

Лимит — 2 400 000 ₽ в год (`NpdIncomeLimit = 2_400_000_00` — значение в копейках). Обработка превышения лимита лежит за пределами текущей реализации `TaxCalculateForSelfEmployed` — функция считает налог на переданный доход без проверки лимита.

### Страховые взносы

При НПД страховые взносы (ПФР, ФОМС, ФСС) не начисляются. В `CalculateMonthlyTax()` для самозанятых блок `calcEmployerContributions()` не вызывается, все взносы остаются нулевыми.

## Ветки расчёта

`calculate.go` → `CalculateMonthlyTax()` - главная функция. Switch по категории налогоплательщика:

| Условие | Функция | Шкала |
|---------|---------|-------|
| Самозанятый (НПД) | `TaxCalculateForSelfEmployed` | 4% или 6% НПД |
| Нерезидент | `TaxCalculateForNotResident` | 30% на всё |
| Льготник (силовые) | `TaxCalculateWithPrivilege` | 13% / 15% |
| Нет РК, нет СН | `TaxCalculateOnlySalary` | 5-ступ. 13–22% |
| Есть РК, нет СН | `TaxCalculateOnlySalary` (оклад с РК) | 5-ступ. 13–22% |
| Есть СН | `TaxCalculateWithNorth` | База A: 5-ступ. + База B: 13%/15% |

После расчёта НДФЛ - одна функция `calcEmployerContributions()` добавляет взносы ПФР/ФОМС/ФСС к каждому месяцу.

**Исключение для НПД и ГПХ:** при НПД взносы не начисляются вовсе (`isSelfEmployed` — пропуск цикла); при ГПХ начисляются ПФР и ФОМС, но `fss = 0`.

## Поток запроса (backend)

```
gRPC-запрос
    ↓
FromPrivateRequest() / FromPublicRequest()  - proto → internal struct
    ↓
ValidateCalculateInput()  - валидация (оклад > 0, диапазоны коэффициентов)
    ↓
cacheGet()  - проверка Redis
    ↓ (промах)
CalculateMonthlyTax()  - расчёт, возвращает []MonthlyTax
    ↓
ToGRPCPrivateResponse() / ToGRPCPublicResponse()  - internal → proto
    ↓
cacheSet()  - сохранение в Redis
    ↓
gRPC-ответ
```

**Private vs Public:** два метода, два набора полей. `CalculatePrivate` возвращает взносы работодателя (ПФР/ФОМС/ФСС), `CalculatePublic` - нет. Логика расчёта одна и та же.


## Правила, которые нельзя нарушать

1. **Бизнес-логика только в `internal/calculate/`** - web-слой не считает налог
2. **Все деньги в копейках, тип `uint64`** - никаких `float64`
3. **Округление только через `RoundTaxAmount()`** - реализует правило 50 копеек (ст. 52 НК РФ)
4. **Изменение proto → запустить `make codegen`** - иначе сгенерированный код устареет
5. **Каждая новая ветка расчёта → unit-тест с граничными значениями**
